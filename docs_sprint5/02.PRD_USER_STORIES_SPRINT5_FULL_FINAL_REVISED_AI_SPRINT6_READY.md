## 1. Executive Summary

Sprint 5 adalah release program untuk mengubah HL Health Companion dari aplikasi pencatatan kesehatan menjadi produk SaaS kesehatan harian yang edukatif, aman secara medis, subscription-ready, dan siap dikembangkan per phase tanpa merusak Sprint 1–4.

Sprint 5 harus dikerjakan bertahap, bukan sebagai satu sprint raksasa. Urutan implementasi final:

```text
1. Sprint 5 Foundation — Commercial Foundation & Admin Core
2. Sprint 5A — Daily Health & Education MVP
3. Sprint 5B — Hydration Tracker
4. Sprint 5C — AI Clinical Infrastructure & Vectorize Foundation
5. Sprint 5D — Cycle Tracking
6. Sprint 5E — Telegram Inline Hydration UX
```

Tujuan utama Sprint 5:

```text
1. Membuat aplikasi siap dijual melalui plan Free, Premium Monthly, Premium 3-Month, Premium Yearly, dan Family Premium.
2. Menambahkan admin area untuk role, permission, subscription, plan, feature, AI config, system config, education content, master data, dan audit.
3. Mengubah dashboard menjadi Daily Health Hub yang edukatif dan mudah dipahami.
4. Menambahkan daily symptom log agar angka pengukuran dan keluhan fisik dapat dibaca bersama.
5. Menambahkan deterministic red flag guardrail untuk gejala berbahaya.
6. Menambahkan hydration tracker dengan target dinamis.
7. Menyiapkan AI Clinical Infrastructure, Vectorize, memory isolation, context retrieval, disclaimer, audit, entitlement, dan feature flag agar Sprint 6 dapat membangun AI Doctor-like Clinical Copilot secara aman.
8. Menambahkan cycle tracking dengan privacy dan contraception guardrail.
9. Menambahkan Telegram inline hydration setelah hydration tracker stabil.
```

### Sprint 5 to Sprint 6 AI Scope Boundary

Sprint 5 tidak membangun AI dokter, AI diagnosis final, AI emergency authority, AI prescription engine, atau AI medication dosage engine. Sprint 5 hanya membangun pondasi teknis dan safety untuk AI yang lebih kuat di Sprint 6.

```text
Sprint 5C = AI Clinical Infrastructure & Vectorize Foundation
Sprint 6 = AI Doctor-like Clinical Copilot / Personal Health Intelligence
```

Sprint 6 yang akan dirancang setelah Sprint 5 dapat mencakup:

```text
AI Symptom Interview
AI Follow-up Question Engine
AI Differential Possibility Builder
AI Risk Explanation
AI Missing Data Detector
AI Doctor Handoff Generator
AI Longitudinal Pattern Reasoning
AI Caregiver Summary
Clinical Safety Evaluation
```

Sprint 6 tetap harus memakai deterministic guardrail untuk emergency, red flag, medication safety, privacy, audit, dan escalation. AI boleh membantu reasoning dan komunikasi, tetapi tidak boleh menjadi otoritas tunggal untuk diagnosis final, emergency final, resep obat, atau perubahan dosis obat.

## 3. Non-Negotiable Product Rules

### 3.1 Rule First, AI Assisted

Semua status kesehatan, severity, emergency level, red flag, dan guardrail harus berasal dari deterministic logic atau rules engine. AI pada Sprint 5 hanya boleh dipakai untuk AI Vision, penjelasan aman, summarization, context retrieval, dan infrastruktur memori. AI Doctor-like Clinical Copilot dipindahkan ke Sprint 6.

```text
measurement / symptom / hydration / cycle data
→ deterministic rule / guardrail
→ status / severity / warning / safety event
→ Sprint 5 AI: assistive explanation, retrieval, trace, infrastructure
→ Sprint 6 AI: clinical copilot experience dengan guardrail dan validation layer
```

### 3.2 Safety First

```text
1. AI tidak boleh menentukan emergency sendirian.
2. AI tidak boleh memberi diagnosis pasti.
3. AI tidak boleh meresepkan obat.
4. AI tidak boleh mengubah dosis obat.
5. AI tidak boleh mengklaim menggantikan dokter.
6. Red flag symptom harus diproses server-side secara deterministic.
7. Cycle guardrail harus blocking, bukan toast.
8. Overhydration warning harus jelas sebagai warning, bukan diagnosis.
9. Disclaimer wajib tampil pada semua output AI medis.
```

### 3.3 Consent First

```text
1. Data user adalah milik user.
2. Caregiver hanya dapat melihat data sesuai permission/consent.
3. Cycle, pregnancy, lactation, menopause, symptom detail, red flag detail, report detail, dan AI memory adalah data sensitif.
4. Support/admin tidak boleh melihat detail medis sensitif secara default.
5. Akses data sensitif oleh internal harus memakai permission khusus dan audit.
```

### 3.4 Commercial Ready

```text
1. Fitur premium harus dikontrol melalui plan, entitlement, dan quota.
2. Admin harus bisa mengatur config non-secret tanpa deploy ulang.
3. Secret tidak boleh disimpan atau ditampilkan sebagai plaintext di D1/UI/API.
4. Semua admin/security-sensitive action harus tercatat di audit log.
```

### 3.5 No Regression

Sprint 5 tidak boleh merusak fitur Sprint 1–4:

```text
existing login
measurement submit
AI Vision
manual override
rule engine
family/caregiver alert
medication
fasting
report/PDF
history
export
PWA
settings
notification foundation
```

## 13. User Stories — Foundation

### US-5F-001 — Global Role untuk User dan Admin

Sebagai pemilik aplikasi, saya ingin sistem memiliki role global agar akses user, support, admin, dan super admin dapat dibedakan dengan aman.

Acceptance Criteria:

```gherkin
Given user berhasil login
When sistem membaca role user
Then user biasa hanya memiliki role user

Given user memiliki role admin
When membuka admin area
Then user dapat mengakses admin dashboard sesuai permission

Given user tidak memiliki akses admin
When membuka admin area
Then sistem menolak akses dengan 403 atau redirect aman
```

### US-5F-002 — Permission-Based Access

Sebagai developer, saya ingin akses admin memakai permission granular agar akses tidak hardcoded berlebihan.

Acceptance Criteria:

```gherkin
Given admin tidak memiliki permission tertentu
When admin memanggil action terkait permission itu
Then sistem mengembalikan forbidden

Given admin memiliki permission yang sesuai
When admin memanggil action terkait
Then request diproses dan audit log tercatat
```

### US-5F-003 — Plan Free dan Premium

Sebagai pemilik aplikasi, saya ingin membuat plan Free, Monthly, 3-Month, Yearly, dan Family agar aplikasi bisa dijual sebagai membership.

Acceptance Criteria:

```gherkin
Given superAdmin membuka Plans & Features

Then plan tersimpan dengan billing interval, harga, status, dan fitur

Given user baru register
When akun dibuat
Then user otomatis memiliki plan Free aktif
```

### US-5F-004 — Feature Entitlement per Plan

Sebagai pemilik aplikasi, saya ingin setiap plan memiliki fitur berbeda agar fitur premium dapat dikontrol.


## 16. PRD — Features

### F5A-001 — Google OAuth

User dapat login/register menggunakan Google.

Rules:

```text
1. Email/password existing tetap berjalan.
2. Scope minimal: openid, email, profile.
3. Email Google harus verified.
4. Akun existing tidak boleh auto-merge hanya berdasarkan email.
5. Jika profile belum lengkap, user diarahkan ke onboarding.
6. User dapat link/unlink Google dari settings sesuai aturan keamanan.
```

# Sprint 5E — Telegram Inline Hydration UX

## 35. PRD — Phase Objective

Membuat user bisa mencatat hidrasi langsung dari Telegram tanpa membuka aplikasi web/PWA.

## 36. PRD — Business Value

```text
1. Meningkatkan retention dan habit.
2. Mengurangi friksi input hidrasi.
3. Memperkuat Telegram integration.
4. Membuat reminder lebih actionable, bukan hanya notifikasi pasif.
```

## 37. PRD — Features

### F5E-001 — Hydration Reminder Cron

Sistem mengirim reminder hydration pada jam operasional yang dikonfigurasi.

### F5E-002 — Telegram Inline Keyboard

Pesan Telegram menampilkan:

```text
Total minum hari ini
Target hari ini
Tombol +200ml
Tombol +600ml
Tombol buka aplikasi
```

### F5E-003 — Secure Callback Processing

Callback harus divalidasi:

```text
webhook secret valid
chatId terhubung ke user
callback data valid
callback tidak duplicate
```

### F5E-004 — Quick Add Water Log

Jika valid:

```text
1. Water log tersimpan.
2. Total hari ini dihitung ulang.
3. Overhydration warning tetap diproses.
4. Telegram message diedit dengan total terbaru.
```

### F5E-005 — Telegram Security Event

Callback invalid, unknown chatId, invalid secret, dan duplicate harus dicatat sesuai severity.

## 38. PRD — Functional Requirements

| ID | Requirement |
|---|---|
| 5E-FR-001 | Cron hanya mengirim reminder ke user yang mengaktifkan hydration reminder. |
| 5E-FR-002 | Cron hanya mengirim ke user yang punya Telegram linked aktif. |
| 5E-FR-003 | Reminder harus menghormati jam operasional user. |
| 5E-FR-004 | Telegram message punya inline button +200ml dan +600ml. |
| 5E-FR-005 | Webhook harus validasi secret. |
| 5E-FR-006 | Webhook harus validasi chatId mapping. |
| 5E-FR-007 | Webhook harus idempotent untuk callback duplicate. |
| 5E-FR-008 | Valid callback membuat water log. |
| 5E-FR-009 | Setelah log, total hidrasi dihitung ulang. |
| 5E-FR-010 | Telegram message diedit dengan total terbaru. |
| 5E-FR-011 | Invalid callback tidak boleh membuat water log. |

## 39. User Stories — Sprint 5E

### US-5E-001 — Kirim Reminder Telegram

Sebagai user, saya ingin menerima pengingat minum dari Telegram.

Acceptance Criteria:

```gherkin
Given hydration reminder enabled
And Telegram linked
When cron berjalan dalam jam operasional
Then user menerima pesan Telegram dengan total dan target hari ini
And pesan memiliki inline keyboard +200ml dan +600ml
```

### US-5E-002 — Tambah Air dari Inline Button

Sebagai user, saya ingin menambah catatan minum langsung dari tombol Telegram.

Acceptance Criteria:

```gherkin
Given user menerima Telegram reminder
When user klik +200ml
Then sistem memvalidasi secret dan chatId
And menambah water log 200ml
And Telegram message diedit dengan total terbaru

When user klik +600ml
Then sistem menambah water log 600ml
```

### US-5E-003 — Tolak Webhook Tidak Valid

Sebagai sistem, saya ingin menolak request Telegram palsu.

Acceptance Criteria:

```gherkin
Given webhook secret tidak valid
When request diterima
Then sistem menolak request
And tidak menambah water log
And security event atau audit log tercatat

Given chatId tidak dikenal
Then request ditolak dan tidak menyimpan log
```

### US-5E-004 — Idempotent Callback

Sebagai sistem, saya ingin mencegah duplicate log jika Telegram retry callback.

Acceptance Criteria:

```gherkin
Given callback id yang sama dikirim dua kali
When webhook diproses
Then hanya satu water log dibuat
And response berikutnya tidak menggandakan total
```
