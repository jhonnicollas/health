# 2C. Ponytail Coding Skill Rule

When coding implementation for any TDD task, agent must follow Ponytail discipline:

```text
1. Check whether the feature/component/service already exists before adding new code.
2. Prefer stdlib/native browser APIs/project utilities before adding new dependencies.
3. Implement the smallest passing change for the failing test.
4. Do not remove validation, security, privacy, error handling, accessibility, or data-loss handling to pass quickly.
5. Do not create large abstraction/framework layers unless required by multiple passing tests.
6. Keep the code easy for the next agent to resume.
```

This rule is implementation discipline, not permission to skip tests.

# 4. Red-Green-Refactor Checklist

For every task:

```text
RED:
[ ] Write failing unit/API/UI/security test first.
[ ] Confirm test fails for the right reason.

GREEN:
[ ] Implement smallest code path.
[ ] Confirm new test passes.

REFACTOR:
[ ] Remove duplication.
[ ] Keep behavior covered.
[ ] Add negative/security/privacy test.

GATE:
[ ] Run task-level command.
[ ] Update WORK_LOG.md.
[ ] Update HANDOFF.md.
```

Write first:

```text
Given user has no admin.access
When GET /api/admin/me
Then response is 403

Given user has admin.access
When GET /api/admin/me
Then response is 200 and returns roles/permissions

Given user role revoked
When GET /api/admin/me
Then response is 403
```
