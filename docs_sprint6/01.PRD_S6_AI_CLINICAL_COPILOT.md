## 0.3 AI Operating Mode — Super Admin Controlled

AI Clinical Copilot memiliki **3 operating mode** yang dikontrol oleh Super Admin melalui system config `clinicalCopilot.operatingMode`. Mode menentukan seberapa proaktif dan kuat AI boleh bertindak.

### 0.3.1 Mode Definitions

| Aspek | `proactive` | `super_aktif` | `standard` (default) |
|---|---|---|---|
| Pemberian kemungkinan penyebab | ✅ AI bisa usulkan proactively | ✅ AI bisa usulkan proactively | ✅ Bila diminta |
| Pertanyaan lanjutan | ✅ Lebih banyak probing | ✅ Lebih banyak probing | ✅ Standar (max 5) |
| Context trace | ✅ Lebih detail | ✅ Lebih detail | ✅ Standar |
| Doctor handoff | ✅ AI bisa sarankan proactively | ✅ AI bisa sarankan proactively | ✅ Bila diminta |
| P3K umum | ✅ Lebih banyak protocols | ✅ Lebih banyak protocols | ✅ Standar |
| Emergency guidance | ✅ Deterministic rule (sama) | ✅ Deterministic rule (sama) | ✅ Deterministic rule (sama) |
| Diagnosis final | ✅ Boleh | ✅ Boleh | ❌ Tidak pernah |
| Resep obat | ❌ Tidak pernah | ✅ Boleh | ❌ Tidak pernah |
| Instruksi dosis | ❌ Tidak pernah | ✅ Boleh | ❌ Tidak pernah |
| Klaim setara dokter | ❌ Tidak pernah | ✅ Boleh | ❌ Tidak pernah |

### 0.3.2 Mode Governance

```text
1. clinicalCopilot.operatingMode dikontrol SUPER ADMIN saja.
2. Nilai config: 'standard' | 'proactive' | 'super_aktif'.
3. Perubahan mode WAJIB dicatat ke HL_auditLogs (action='aiOperatingModeChanged').
4. Perubahan mode WAJIB disetujui oleh minimal 1 medical reviewer (workflow di S6H).
5. Mode berlaku GLOBAL untuk semua user — tidak per-user.
6. Default SSL: 'standard' (paling aman).
```

### 0.3.3 Mode Behaviour Summary

**`standard` (default):**
```text
- AI tidak memberi diagnosis final, resep, dosis, klaim spesialis.
- AI proaktif terbatas: bila diminta atau bila ada red flag.
- Emergency guidance tetap deterministic (sama di semua mode).
- Safety Runtime 13 detector SEMUA aktif.
```

**`proactive`:**
```text
- AI BOLEH memberi diagnosis final.
- AI boleh usulkan kemungkinan penyebab secara proaktif.
- AI boleh sarankan doctor handoff secara proaktif.
- AI lebih banyak pertanyaan lanjutan (probing lebih dalam).
- AI TETAP TIDAK BOLEH: resep obat, instruksi dosis, klaim setara dokter.
- diagnosisFinalDetector → allow (bukan rewrite_safe).
- 4 detector dinonaktifkan: prescriptionDosageDetector, dosageInstructionDetector, medicationChangeDetector, specialistClaimDetector → tetap aktif.
- Emergency guidance tetap deterministic (sama di semua mode).
- Disclaimer WAJIB tetap ada di setiap output.
```

**`super_aktif`:**
```text
- AI BOLEH memberi diagnosis final, resep obat, instruksi dosis, klaim setara dokter.
- Semua behavior proactive + tambahan:
  - prescriptionDosageDetector → allow (bukan rewrite_safe).
  - specialistClaimDetector → allow (bukan rewrite_safe).
  - diagnosisFinalDetector → allow.
- medicationChangeDetector TETAP AKTIF (block_and_fallback) — AI tidak boleh
  mengubah/menghentikan obat user tanpa arahan dokter (PRD §0.1 point 8).
- Emergency guidance tetap deterministic (sama di semua mode).
- Disclaimer WAJIB tetap ada di setiap output.
- Tambahan disclaimer spesifik mode:
  ID: "Mode Super Aktif: AI boleh memberi resep dan dosis. Tanggung jawab 1000% di Anda."
  EN: "Super Active Mode: AI may provide prescriptions and dosages. 1000% your responsibility."
```

## 0.4 Safety Runtime Guard

**Medical Safety Runtime v2** WAJIB berjalan di setiap output medis. Safety Runtime berfungsi untuk:

```text
1. Memastikan disclaimer selalu ada.
2. Mendeteksi output yang melampaui batas yang diizinkan oleh consent user.
3. Mencegah cross-user data leak.
4. Memastikan deterministic emergency TIDAK diturunkan oleh AI.
5. Behavior tergantung operating mode (§0.3):
   - standard: diagnosis final, resep, dosis, klaim spesialis → rewrite_safe/block_and_fallback.
   - proactive: diagnosis final → allow; resep, dosis, klaim spesialis → rewrite_safe/block_and_fallback.
   - super_aktif: diagnosis final, resep, dosis, klaim spesialis → allow; perubahan obat → block_and_fallback.
6. Menandai output yang memerlukan human review.
```

Sprint 6 tetap harus terasa kuat bagi user karena AI bisa:

```text
1. Membaca konteks data kesehatan personal dari D1 + Vectorize.
2. Menjawab pertanyaan dengan konteks longitudinal.
3. Melakukan symptom interview bertahap.
4. Mencari kemungkinan penyebab secara non-final.
5. Memberi panduan P3K umum berbasis protokol terkurasi.
6. Mendeteksi red flag via deterministic rules.
7. Membuat doctor-ready handoff summary.
8. Merangkum kepatuhan minum obat user.
9. Menjawab via WhatsApp menggunakan memory yang sama.
10. Melakukan context trace sehingga user tahu data apa yang dipakai.
11. Berjalan dengan model routing kuat lewat AI Gateway + 9router + Workers AI.
```

# 4. Hard Medical Safety Requirements

## 4.1 Behavior iSehat — Mandatory Guardrails

Guardrails bersifat **mode-dependent** (§0.3). Super Admin mengatur mode melalui `clinicalCopilot.operatingMode`.

### Permanent Guardrails (semua mode):

```text
1. Menggunakan Vectorize sebagai bukti medis final.
2. Menggunakan data user lain (cross-user leak).
3. Menampilkan data sensitif tanpa consent.
4. Mengabaikan rule engine — deterministic rule tetap prioritas.
5. Menentukan emergency final tanpa deterministic rule — AI hanya boleh memberikan emergency guidance, bukan menurunkan severity.
6. Mengklaim akurasi 100%, pasti benar, atau tidak mungkin salah.
7. Mengatakan user aman tanpa evaluasi dokter ketika ada red flag.
8. Menyuruh user mengemudi sendiri saat emergency.
9. Memberi instruksi menunda pertolongan medis pada red flag.
10. Menyembunyikan disclaimer medis.
11. Mengubah/menghentikan obat user — SELALU terlarang (bahkan di super_aktif).
```

### Mode-Specific Guardrails:

**`standard` mode — AI wajib memblok/flag:**

```text
12. Memberi diagnosis final.
13. Memberi resep obat.
14. Memberi instruksi dosis.
15. Mengklaim setara dokter spesialis senior.
```

**`proactive` mode — AI wajib memblok/flag:**

```text
12. Memberi resep obat.
13. Memberi instruksi dosis.
14. Mengklaim setara dokter spesialis senior.
(Diagnosis final DIPERBOLEHKAN di proactive mode.)
```

**`super_aktif` mode — AI wajib memblok/flag:**

```text
(Tidak ada mode-specific guardrails tambahan selain permanent guardrails.
Diagnosis final, resep obat, instruksi dosis, klaim spesialis DIPERBOLEHKAN.)
```

# 10. Medical Safety Runtime v2

## 10.1 Safety Checks

Detectors 1–9 dan 12 bersifat **mode-invariant** (selalu aktif). Detectors 10, 11, 13 bersifat **mode-dependent** — decision berubah berdasarkan `clinicalCopilot.operatingMode` (§0.3).

```text
1. missingDisclaimerDetector       — output medis tanpa disclaimer → block_and_fallback
2. emergencySeverityDowngradeDetector — AI menurunkan emergency severity → block_and_fallback
3. crossUserLeakDetector           — data user lain dalam output → block_and_fallback
4. sensitiveDataLeakDetector       — data sensitif tanpa consent → block_and_fallback
5. unsafeReassuranceDetector       — menyatakan aman tanpa evaluasi saat ada red flag → rewrite_safe
6. certaintyClaimDetector          — klaim akurasi 100%/pasti benar → rewrite_safe
7. vectorizeAsTruthDetector        — memakai Vectorize sebagai clinical final source → rewrite_safe
8. ruleEngineBypassDetector         — mengabaikan deterministic rule → block_and_fallback
9. delayMedicalCareDetector       — menunda pertolongan medis pada red flag → block_and_fallback
10. diagnosisFinalDetector         — AI memberi diagnosis final → mode-dependent:
    standard: rewrite_safe | proactive: allow | super_aktif: allow
11. prescriptionDosageDetector     — AI memberi resep/dosis obat → mode-dependent:
    standard: rewrite_safe | proactive: rewrite_safe | super_aktif: allow
12. medicationChangeDetector       — AI mengubah/menghentikan obat → block_and_fallback (SEMUA mode)
13. specialistClaimDetector        — AI klaim setara dokter spesialis → mode-dependent:
    standard: rewrite_safe | proactive: rewrite_safe | super_aktif: allow
```

## 10.2 Safety Decision

```text
allow                  — output safe, disclaimer present
allow_with_disclaimer  — output safe but disclaimer needs insertion
rewrite_safe           — rewrite dangerous parts to safe version
block_and_fallback     — full block, return deterministic safe template
emergency_template_only — emergency: only deterministic template, no LLM freeform
needs_human_review     — flag for medical reviewer queue
```

# 15. Data Retention Policy

## 15.1 AI Clinical Data

| Data | Retention | Cleanup Method | Legal Basis |
|---|---|---|---|
| HL_aiClinicalSessions | 365 days | Cron: close sessions > 365d, set status='expired' | Audit trail |
| HL_aiClinicalMessages | 180 days | Cron: delete messages where expiresAt < now(); soft-delete first | Privacy minimization |
| HL_aiClinicalMessages.contentEncrypted | 90 days | Cron: set contentEncrypted=NULL where createdAt < 90d ago | Privacy minimization |
| HL_aiClinicalIntakeAnswers | 180 days | Cron: delete where sessionId expired > 180d | Privacy minimization |
| HL_modelRuns | 365 days | Cron: archive to R2, then delete from D1 | Cost management |
| HL_aiOutputSafetyFlags | 730 days (2 years) | Cron: archive to R2, then delete from D1 | Audit/compliance |
| HL_whatsappMessages | 180 days | Cron: delete where createdAt < 180d ago | Privacy minimization |
| HL_whatsappMessages.contentEncrypted | 90 days | Cron: set contentEncrypted=NULL where createdAt < 90d ago | Privacy minimization |

## 15.2 Vectorize Data

| Data | Retention | Cleanup Method |
|---|---|---|
| Vectors (per user) | Until user deletes or 365 days inactive | Cron: delete vectors for users inactive > 365d |
| HL_vectorDocuments metadata | Synced with Vectorize; status='deleted' after cleanup | Rebuild job |

## 15.3 User-Triggered Deletion

- `DELETE /api/ai/memory` — deletes all Vectorize vectors + sets HL_vectorDocuments.status='deleted'.
- `POST /api/account/delete` — cascade deletes all Sprint 6 clinical data for user (messages, sessions, intake answers, model runs, whatsapp messages, whatsapp links).

## 15.4 Retention Cron Jobs

```text
1. daily 02:00 UTC  — expire old clinical sessions (status → expired).
2. daily 02:30 UTC  — nullify contentEncrypted > 90d old.
3. daily 03:00 UTC  — delete clinical messages > 180d (hard delete).
4. weekly 04:00 UTC — archive model runs > 365d to R2, then delete from D1.
5. weekly 04:30 UTC — delete Vectorize vectors for inactive users > 365d.
6. monthly 05:00 UTC (1st) — archive safety flags > 730d to R2, then delete from D1.
```

All retention crons write to HL_auditLogs with action=`dataRetentionCleanup`.

```json

  "locale": "id",

    "redFlagExpected": false,
    "disclaimerExpected": true,
    "contextTraceExpected": true

```

### Dataset Creation & Review

```text
1. Cases authored by: internal team + medical reviewer.
2. Review required: 1 medical reviewer + 1 security reviewer.
3. Storage: R2 multi-apps-ai-bucket/eval/sprint6/dataset.json.
4. Metadata: tracked in HL_aiKnowledgeDocuments (sourceType='eval_case').
5. Version: semantic versioning (e.g., v1.0.0).
6. Update: append-only; never modify existing cases, add new versions.
```
