## 1. Executive Summary

Sprint 5 adalah release program besar untuk menjadikan HL Health Companion siap menjadi produk SaaS kesehatan harian yang dapat dijual, aman secara medis, mudah digunakan, dan edukatif.

Sprint 5 tidak boleh dikerjakan sebagai satu sprint raksasa. Sprint 5 harus dikerjakan bertahap agar setiap bagian bisa diuji, tidak merusak Sprint 1–4, dan tidak mencampur fitur kesehatan sensitif dengan fondasi komersial.

Sprint 5 dibagi menjadi enam phase:

| Phase | Nama | Fokus | Prioritas |
|---|---|---|---|
| Sprint 5 Foundation | Commercial Foundation & Admin Core | RBAC, subscription, entitlement, admin, audit, config | P0 |
| Sprint 5A | Daily Health & Education MVP | Google OAuth, Daily Health Hub, education layer, symptom log, red flag | P0 |
| Sprint 5B | Hydration Tracker | Target air dinamis, widget, quick input, warning | P1 |
| Sprint 5C | AI Clinical Infrastructure & Vectorize Foundation | AI config, safety runtime, memory/indexing pipeline, internal context retrieval, Sprint 6 AI Clinical Copilot readiness | P1 |
| Sprint 5D | Cycle Tracking | Cycle tracking wanita, privacy, guardrail kontrasepsi | P2 |
| Sprint 5E | Telegram Inline Hydration UX | Reminder dan quick add hidrasi via Telegram | P2 |

Tujuan utama Sprint 5:

```text
1. Menjadikan aplikasi siap dijual dengan plan Free, Monthly, 3-Month, Yearly, dan Family.
2. Menambahkan admin area untuk mengelola user, role, subscription, plan, feature, AI config, content edukasi, dan audit.
3. Mengubah aplikasi dari sekadar pencatat angka menjadi daily health companion yang edukatif.
4. Menambahkan pencatatan keluhan harian agar data objektif dan subjektif user bisa dibaca bersamaan.
5. Menambahkan red flag guardrail deterministic agar kondisi berisiko tidak bergantung pada AI.
6. Menambahkan hydration tracker sebagai habit harian.
7. Membangun AI Clinical Infrastructure agar Sprint 6 dapat menghadirkan AI Doctor-like Clinical Copilot yang kuat, aman, terukur, dan tidak halusinatif.
8. Menambahkan cycle tracking secara aman, private, dan tidak menyesatkan sebagai alat kontrasepsi.
9. Menambahkan Telegram inline action setelah hydration web/PWA stabil.
```

## 2. Product Background

Sprint 1–4 sudah menyelesaikan fondasi utama aplikasi:

```text
- Health measurement logging.
- AI Vision extraction dari gambar alat kesehatan.
- Manual override untuk hasil AI Vision.
- Rule-based interpretation.
- Family/caregiver flow.
- Emergency notification foundation.
- Medication tracking.
- Fasting tracking.
- Report dan doctor-ready PDF.
- Pattern detection.
- PWA.
- Export dan privacy foundation.
```

Namun, aplikasi belum siap sebagai produk komersial penuh karena masih ada gap:

```text
1. Belum ada role global aplikasi untuk user, support, admin, dan super admin.
2. Belum ada subscription plan dan entitlement untuk membedakan Free vs Premium.
3. Belum ada admin panel terpusat untuk mengelola konfigurasi aplikasi.
4. Belum ada sistem edukasi konsisten yang menjelaskan arti, tujuan, manfaat, dan tindakan dari setiap pengukuran.
5. Keluhan subjektif user belum menjadi data terstruktur.
6. Hidrasi belum menjadi tracker harian.
7. AI belum memiliki clinical infrastructure yang cukup kuat untuk menjadi AI Doctor-like Clinical Copilot pada Sprint 6.
8. Cycle tracking belum tersedia dan perlu privacy/medical guardrail ketat.
9. Telegram quick action belum tersedia.
```

## 4. Non-Negotiable Product Rules

### 4.1 Rule First, AI Assisted, Clinical-Copilot Ready

Semua status kesehatan, severity, emergency level, red flag, hydration warning, cycle guardrail, entitlement, dan quota harus berasal dari deterministic logic atau rules engine. AI Sprint 5 tidak boleh menjadi penentu klinis final. AI Sprint 5 bertugas menyiapkan clinical context infrastructure untuk Sprint 6.

```text
Measurement / symptom / hydration / cycle data
→ deterministic rule / guardrail
→ status / severity / warning / safety event
→ structured context package
→ AI boleh membantu narasi, insight aman, dan preparation untuk Sprint 6 Clinical Copilot
```

Sprint 6 dapat membangun AI Doctor-like Clinical Copilot di atas pondasi ini, tetapi tetap harus memakai rule/guardrail sebagai authority untuk emergency escalation dan safety boundary.

### 4.2 Safety First

```text
1. AI tidak boleh menentukan emergency sendirian.
2. AI tidak boleh memberi diagnosis pasti.
3. AI tidak boleh meresepkan obat.
4. AI tidak boleh mengubah dosis obat.
5. AI tidak boleh mengklaim menggantikan dokter.
6. Red flag symptom harus diproses server-side secara deterministic.
7. Cycle guardrail harus blocking, bukan toast.
8. Overhydration warning harus jelas sebagai warning, bukan diagnosis.
9. Disclaimer wajib tampil pada semua output AI medis.
10. Fitur AI Doctor-like Clinical Copilot, AI symptom interview, differential possibility builder, dan clinical reasoning agent dipindahkan ke Sprint 6, bukan bagian dari scope user-facing Sprint 5.
```

### 4.3 Consent First

```text
1. Data user hanya milik user.
2. Caregiver hanya dapat melihat data sesuai consent/permission.
3. Cycle, pregnancy, lactation, menopause, symptom detail, dan AI memory adalah data sensitif.
4. Support/admin tidak boleh melihat detail medis sensitif secara default.
5. Akses data sensitif oleh internal harus memakai permission khusus dan audit.
```

### 4.4 Commercial Ready

```text
1. Semua fitur premium harus bisa dikontrol melalui plan dan entitlement.
2. Usage quota harus bisa diterapkan untuk fitur mahal seperti AI, OCR, PDF, Telegram, dan AI Memory.
3. Admin harus bisa mengatur config non-secret tanpa deploy ulang.
4. Secret tidak boleh disimpan atau ditampilkan sebagai plaintext di D1/UI.
5. Semua admin action harus tercatat di audit log.
```

### 4.5 No Regression

Sprint 5 tidak boleh merusak fitur Sprint 1–4. Existing login, dashboard, measurement submit, AI Vision, rule engine, family alert, medication, fasting, report, PDF, export, dan PWA harus tetap berjalan.

# Sprint 5A — Daily Health & Education MVP

## 17. Phase Objective

Membuat aplikasi lebih edukatif dan manusiawi: user tidak hanya mencatat angka, tetapi memahami arti data dan dapat mencatat keluhan fisik harian.

## 18. Business Value

```text
1. Meningkatkan trust user karena aplikasi menjelaskan arti angka.
2. Menghubungkan data objektif pengukuran dengan keluhan subjektif.
3. Membuat red flag safety flow lebih kuat.
4. Mempermudah onboarding dengan Google OAuth.
5. Membuat dashboard lebih relevan sebagai Daily Health Hub.
```

## 19. Features

### F5A-001 — Google OAuth

User dapat login/register menggunakan Google.

Rules:

```text
1. Email/password existing tetap berjalan.
2. Scope minimal: openid, email, profile.
3. Email Google harus verified.
4. Akun existing tidak boleh auto-merge hanya berdasarkan email.
5. Jika profile belum lengkap, user diarahkan ke onboarding.
6. User dapat link/unlink Google dari settings sesuai aturan keamanan.
```

### F5A-002 — Daily Health Hub

Dashboard Today menjadi pusat ringkasan kesehatan harian.

Konten:

```text
Status hari ini
Pengukuran terakhir
Keluhan aktif hari ini
Quick action catat keluhan
Hydration widget placeholder atau real widget jika 5B aktif
Education card hari ini
AI insight ringkas jika entitlement tersedia
Safety warning jika ada
```

### F5A-003 — Education Layer

Setiap metric/fitur penting harus punya education card.

Isi edukasi:

```text

Manfaat mencatat?
Cara ukur yang benar?
Angka normal/warning/bahaya secara umum

```

Education harus muncul melalui:

```text

first-time guidance
metric detail

report explanation link
```

### F5A-004 — Proactive Symptom Prompt

Jika user submit measurement dengan severity abnormal, UI menanyakan apakah user memiliki keluhan fisik.

Prompt:

```text

```

CTA:

```text

Tidak Ada Keluhan
Lihat Arti Angka Ini
```

### F5A-005 — Daily Symptom Log

Form keluhan harian harus cepat dan mobile-friendly.

Fields:


### F5A-006 — VAS Pain Scale

VAS label:

```text
1–3: Ringan
4–6: Sedang
7–10: Berat
```

UI tidak boleh hanya memakai warna. Harus ada label text.

### F5A-007 — Red Flag Interceptor

Keyword red flag awal:

```text

sesak napas
sesak nafas
kaku kuduk
kelemahan sesisi
pingsan
pandangan gelap

```

Jika red flag terdeteksi:

```text

3. Emergency/caregiver notification existing dapat dipicu jika consent tersedia.
4. UI menampilkan Emergency Blocking UI.
5. User tidak langsung diarahkan ke dashboard.
6. AI tidak boleh menjadi penentu emergency.
```

History/timeline harus mendukung row:

```text

symptom
hydration setelah 5B

safety event
alert existing
```
