# 1. Purpose

This SRS defines software requirements for building **iSehat Content Engine**, an internal AI-assisted content production tool for iSehat.

CE-1 focuses only on generating, reviewing, safety-checking, approving, and exporting text-based social content.

The purpose of this document is to give an AI coding agent a precise, bounded, implementation-ready requirement set.

# 2. Scope

## 2.1 Product Scope

**iSehat Content Engine** is an internal growth tool for iSehat. It helps internal users create social media content from brand memory, campaign planning, idea generation, draft generation, safety checking, review, approval, and Markdown/Text export.

This product is **not** the public iSehat health app. It is not used by iSehat end users.

# 3. Core Principles

```text
1. AI drafts.
2. System classifies.
3. System checks safety.
4. Human approves exact revision.
5. Only approved current revision can be final-exported.
6. Everything sensitive is audited.
7. D1 is the CE-1 source of truth.
8. CE-1 must run without R2, Vectorize, Queues, Workflows, OAuth, or renderer.
9. Secrets are never stored plaintext in D1 or returned to frontend.
10. AI prompt context must be whitelist-based.
```

# 6. System Constraints

## 6.1 Technical Constraints

```text
1. Backend target: Cloudflare Workers.
2. API framework: Hono or equivalent Worker-compatible framework.
3. Database: Cloudflare D1.
4. CE-1 must not require R2.
5. CE-1 must not require Vectorize.
6. CE-1 must not require platform OAuth.
7. CE-1 must not use ffmpeg/Remotion/video rendering.
8. AI provider must be configurable and mockable.
9. All protected endpoints must enforce server-side permission guard.
10. All AI jobs must be quota-checked before execution.
11. CE-1 AI jobs are synchronous-with-job-log, not background queue jobs.
```

## 6.2 Safety Constraints

```text
1. AI output must never be treated as final without safety report and human approval.
2. Every draft revision must receive a safety report row.
3. Non-health content receives safetyReport with safetyStatus=safe and note="Non-health content; medical safety check not required."
4. Uncertain health classification is treated as health_content.
5. Safety report must match current draft revision before approval.
6. Approval must match current draft revision before final export.
7. Blocked content cannot be approved or final-exported.
8. Warning content requires explicit reviewer override with reason.
9. Any edit after approval resets approval and safety status.
10. If sourceTraceRequired=true, approval requires at least one source reference for same draftId + revisionNumber.
```

## 6.3 Security Constraints

```text
1. Secrets must not be stored plaintext in D1.
2. API responses must never include raw API keys, access tokens, refresh tokens, webhook secrets, or secret values.
3. Logs must redact Authorization, access_token, refresh_token, api_key, secret, password, client_secret, webhook_secret, and token-like fields.
4. Audit logs must not store raw secrets.
5. AI prompt context builder must use whitelist fields only.
6. External/retrieved content must be treated as untrusted context.
7. Full config rows must not be passed to AI prompts.
```

## 8.10 Medical Safety Rules

Blocked content:

```text
1. Final diagnosis claim.
2. Doctor replacement claim.
3. Prescription or dosage instruction.
4. Emergency authority claim.
5. Guaranteed outcome.
6. 100% accuracy claim.
7. Misleading medical device claim.
8. Before-after medical outcome promise.
9. Disease prevention/cure claim without strong source.
```

Warning content:

```text
1. Fearmongering.
2. Excessive medical clickbait.
3. Ambiguous diagnosis-like wording.
4. Specific medical factual claim without source trace.
5. Risk statement without enough context.
```

Allowed content:

```text
1. iSehat helps users record health data.
2. iSehat helps users understand health history.
3. iSehat helps prepare doctor-ready reports.
4. AI assists input/explanation, not final diagnosis.
5. Manual override remains available.
6. Rule-based safety warning helps awareness.
7. Content is educational, not medical consultation.
```

Required disclaimer for health content:

```text
Konten ini bersifat edukatif dan bukan pengganti konsultasi medis. Jika mengalami keluhan berat, memburuk, atau kondisi darurat, segera hubungi tenaga medis.
```

# 9. State Machine Requirements

## 9.1 IdeaStatus

```text
idea
idea_approved
rejected
archived
```

Allowed transitions:

| From | To | Condition |
|---|---|---|
| idea | idea_approved | owner/marketingAdmin approves idea |
| idea | rejected | owner/marketingAdmin rejects idea |
| idea_approved | archived | authorized archive |
| rejected | archived | authorized archive |

## 9.2 DraftStatus

```text
draft_generating
draft_ready
safety_checking
safety_safe
safety_warning
safety_blocked
needs_review
revision_requested
approved
exported
archived
failed
```

Allowed transitions:

| From | To | Condition |
|---|---|---|
| draft_generating | draft_ready | AI generation success + revision #1 created |
| draft_generating | failed | AI generation failed after max attempts |
| draft_ready | safety_checking | safety-check requested |
| safety_checking | safety_safe | safety report safe |
| safety_checking | safety_warning | safety report warning |
| safety_checking | safety_blocked | safety report blocked |
| safety_safe | needs_review | ready for reviewer |
| safety_warning | needs_review | requires owner/medicalReviewer override |
| safety_blocked | revision_requested | reviewer/system requests rewrite |
| needs_review | approved | authorized reviewer approves current revision |
| needs_review | revision_requested | reviewer requests changes |
| needs_review | archived | authorized archive |
| revision_requested | draft_ready | edit creates new revision |
| approved | draft_ready | edit after approval resets approval |
| approved | exported | final export performed |
| exported | draft_ready | edit after export creates new revision |

Rules:

```text
1. approved can only be reached from needs_review.
2. exported can only be reached from approved.
3. safety_blocked cannot be approved.
4. Any edit after approved/exported resets status to draft_ready.
5. Any new revision requires new safety report before approval/export.
6. Current revision must match latest safety report and approval before final export.
```

```json
  }
```


### Generate Draft

```text
POST /api/content/ideas/:id/generate-draft
Idempotency-Key required
Requires idea.status=idea_approved
Creates revision #1
```

### Safety Check

```text
POST /api/content/drafts/:id/safety-check
Idempotency-Key required
Runs health classifier internally
Creates exactly one conSafetyReports row for current revision
Sets draft status:
- safe => needs_review
- warning => needs_review
- blocked => safety_blocked
```

### Approve Draft

Rules:

```text
1. request revisionNumber must equal draft.currentRevision.
2. safety report must exist for draftId + revisionNumber.
3. if sourceTraceRequired=1, at least one source reference must exist.
4. health content requires owner or medicalReviewer.
5. non-health content requires owner or marketingAdmin.
6. blocked cannot approve.
7. warning requires override reason and owner/medicalReviewer.
```

### Export Markdown

Filename:

```text
{YYYY-MM-DD}_{platform}_{pillarSlug}_{contentFormat}_{draftId}_rev{revision}.md
```

Example:

```text
2026-07-01_instagram_health-data-awareness_carousel_draft_001_rev2.md
```

# 12. AI Requirements

## 12.1 AI Job Mode

CE-1 AI jobs are synchronous-with-job-log:

```text
1. API creates conAiGenerationJobs row.
2. API runs AI call in request lifecycle.
3. API parses and validates JSON.
4. API saves output.
5. API logs usage.
6. API returns completed result.
7. If timeout/failure, job status=failed.
8. Background Queues/Workflows are deferred to CE-2+.
```

## 12.2 Prompt Context Builder

Allowed fields:

```text
1. brand.positioning
2. brand.productValueJson
3. brand.targetAudienceJson
4. brand.tone
5. brand.forbiddenClaimsJson
6. brand.allowedClaimsJson
7. brand.disclaimerTemplate
8. active pillars
9. campaign objective
10. platform
11. format
12. language
```

Forbidden fields:

```text
1. API keys
2. secretRef unless needed for internal routing only, never in prompt
3. raw config rows
4. auth/session data
5. debug logs
6. private user medical data
7. token-like values
```

## 12.3 Output Validation

All AI output must pass:

```text
1. JSON parse.
2. Schema validation.
3. Enum validation.
4. Required field validation.
5. Length validation.
6. Forbidden claim pre-scan.
```
